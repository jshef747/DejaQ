DejaQ — How It Works
01

What DejaQ Does

You built an AI chatbot. Every question costs money. DejaQ is the layer that makes most questions free.

The problem: every question has a price tag

When your app sends a question to an AI like GPT or Claude, you pay for every word — in both the question and the answer. This is called an API call, and they add up fast.

Here's the thing: many questions users ask are nearly identical. "What's the capital of France?" and "hey what's france's capital lol?" are the same question in different clothes. Why pay twice?

💸
Without DejaQ
Every question hits the expensive external API. 100 users asking similar things = 100 API calls billed to you.
🎯
With DejaQ
The first question pays. Similar questions get the cached answer — adjusted to match their tone. Essentially free.
🧠
The Trick
It stores answers by meaning, not exact words. "Close enough" counts as a cache hit.

💡 Aha! — Where the name comes from

DejaQ is a riff on "déjà vu" — that feeling you've seen something before. That's exactly what the semantic cache does: it recognizes when it's "seen" a question before, even phrased completely differently.

The second time someone asks: the cache hit

Next time someone asks "what makes things fall?" — a completely different phrasing but the same meaning — the journey is much shorter:

1

Enrich + Normalize

Same two steps as before — resolve any context references, then extract the core meaning as a clean search key.

2

Cache HIT — no LLM needed

ChromaDB finds the stored answer within a cosine distance of 0.15. The expensive AI call is skipped entirely.

3

Tone Adjustment

The stored answer is neutral ("Gravity is a fundamental force..."). A small local model rewrites it to match the user's casual tone before sending.

Why store a neutral version instead of the original answer?

If User A asked formally and User B asks casually, you need one stored answer that works for both. The neutral version is the master copy — a separate model adds the right tone on the way out for each user.

Models load once and stay

Each AI model takes several seconds to load from disk. DejaQ uses a singleton pattern — the model loads on first use and stays in memory. Every request after that is instant.

CODE — model_loader.py 
class ModelManager:
    _llama = None

    @classmethod
    def load_llama(cls):
        if cls._llama is None:
            cls._llama = Llama.from_pretrained(
                repo_id="hugging-quants/Llama-3.2-1B-Instruct-Q8_0-GGUF",
                filename="*q8_0.gguf",
                verbose=False,
                n_ctx=2048
            )
        return cls._llama
PLAIN ENGLISH

ModelManager is a class with "drawers" — one per model, starting empty.

The Llama drawer starts empty (None).

load_llama() can be called from anywhere, any number of times.

First time: the drawer is empty, so we load the model.

Download Llama 3.2 1B from HuggingFace — it's compressed in GGUF format to run fast on regular hardware.

The wildcard (*) finds the right file. Q8_0 = highest quality that fits in RAM.

2048 = how many words of conversation the model can "hold in mind" at once.

Return the model — whether freshly loaded or already sitting in the drawer from last time.

🔑 The Singleton Pattern

"Check if it exists → create if not → always return the same one." This is one of the most common patterns in production software. When you tell AI to "add a shared resource that's expensive to initialize," this is what it should write.

03

The Cache Miss Journey

When DejaQ can't find a match, it has to think from scratch. Here's every step in that process.

Step 1 — making follow-ups searchable

Problem: users have conversation history. They say "tell me more about it" without saying what "it" is. Before anything else can happen, the Context Enricher resolves these references into full, self-contained questions.

CODE — context_enricher.py 
def enrich(self, message, history):
    if not history:
        return message

    recent_history = history[-6:]

    output = self.llm.create_chat_completion(
        messages=[
          {"role": "system", "content":
            "Rewrite the follow-up into a standalone question..."},
          # few-shot examples...
          {"role": "user", "content":
            f"HISTORY:\n{context_block}\n\nFOLLOW-UP: {message}"},
        ],
        max_tokens=64,
        temperature=0.0
    )
PLAIN ENGLISH

enrich() takes the new message and all prior conversation turns.

No history? The question is already standalone — skip enrichment and return it unchanged.

Only look at the last 6 messages (3 back-and-forths) — older context is usually irrelevant.

Ask a small AI model (Qwen 0.5B) to rewrite the vague follow-up into a full question.

Show it examples first (few-shot) — don't just describe the task, demonstrate it.

Pass in the real conversation history and the vague follow-up message.

64 tokens max — the rewrite should be short and focused.

Temperature 0 = always give the most deterministic answer. Zero creativity — just resolve the reference.

Step 2 — boiling it down to its essence

The enriched question goes to the Normalizer. Its job: strip all personality and extract just the core topic — this becomes the cache key used to search ChromaDB.

What you typed What the Normalizer extracts
"yo explain gravity like im 5 lol""what is gravity"
"could you please describe photosynthesis in detail""how does photosynthesis work"
"hey what's France's capital city?""capital of france"
"I was wondering if you could explain how computers work""how does a computer work"
🗝️ Why normalize before searching?

The cache uses vector embeddings — numbers that represent meaning. Tone, filler words, and slang add noise that pushes embeddings apart even when the meaning is identical. Normalization removes that noise before the comparison.

Step 3 — easy or hard?

On a cache miss, the question needs an actual answer. But which AI should answer it? NVIDIA's DeBERTa model scores the question's complexity across six dimensions and returns a number from 0 to 1.

CODE — classifier.py 
def predict_complexity(self, query):
    encoded = self._tokenizer(
        [f"Prompt: {query}"],
        return_tensors="pt",
        max_length=512,
        truncation=True,
    ).to(self._device)

    with torch.no_grad():
        result = self._model(encoded)

    score = result["prompt_complexity_score"][0]
    complexity = "hard" if score >= 0.4 else "easy"
    return {"complexity": complexity, "score": score}
PLAIN ENGLISH

Given a question string, run it through the complexity classifier.

Convert the text into numbers (tokens) that the neural network can read.

"pt" = PyTorch format. "to(device)" = run on GPU if available, CPU otherwise.

Truncate at 512 tokens — the model has a limit, and most questions are well under it.

no_grad() tells PyTorch we're just predicting, not learning — saves memory and runs faster.

Run the DeBERTa model — it scores creativity, reasoning, domain knowledge, constraints, and more.

The final score blends all dimensions: 35% creativity + 25% reasoning + 15% constraints + ...

Score ≥ 0.4 → "hard" (needs expensive model). Below → "easy" (local Llama handles it).

Return both the label and the raw score so the router and response metadata have full info.

Step 4 — should we even cache this?

After the LLM answers, DejaQ asks: is this response worth storing? Caching a reply to "ok" or "lol" wastes space and pollutes the database. The Cache Filter runs three quick rules.

CODE — cache_filter.py 
def should_cache(enriched_query, normalized_query):
    word_count = len(normalized_query.split())
    if word_count < 3:
        return False, f"query too short ({word_count} words)"

    stripped = enriched_query.strip().rstrip("?.,!")
    if FILLER_PATTERNS.match(stripped):
        return False, "conversational filler"

    enriched_words = len(enriched_query.split())
    if enriched_words < 3:
        return False, "enriched query too vague"

    return True, "passed"
PLAIN ENGLISH

Returns True/False plus a reason string — useful for logging and UI badges.

Rule 1: Count the words in the cleaned-up (normalized) query.

Fewer than 3 words after cleaning? Too vague to be a useful cache entry. Skip.

Rule 2: Strip punctuation from the enriched query, then check it against filler words.

If the whole message was "ok", "thanks", "lol", "haha" — it's social chat, not a question. Skip.

Rule 3: Even after enrichment, if the query is still under 3 words, it's too vague. Skip.

All 3 rules passed — return True so the router knows to kick off background storage.

04

The Cache Hit Magic

How DejaQ finds answers that were never stored verbatim — and serves them with the right personality.

A library that finds books by topic, not title

Traditional caches require an exact match. ChromaDB works differently — it converts every stored question into a list of numbers called an embedding. Questions with similar meanings produce similar numbers, so they land close together in mathematical space.

"What is gravity?" and "explain gravitational force" produce embeddings just 0.08 apart. "How does photosynthesis work?" lands far away. The threshold: distance ≤ 0.15 counts as a hit.

Stored question: "what is gravity"

"explain gravitational force"
92% similar · dist 0.08
✓ HIT
"yo whats gravity lol"
87% similar · dist 0.13
✓ HIT
"how does photosynthesis work"
22% similar · dist 0.78
✗ MISS
"explain Newton's law of gravitation"
84% similar · dist 0.16
✗ MISS*

* dist 0.16 just misses the 0.15 threshold — unless this entry has a high feedback score, which relaxes the limit to 0.20.

How the lookup actually works in code

CODE — memory_chromaDB.py 
def check_cache(self, normalized_query):
    results = self._collection.query(
        query_texts=[normalized_query],
        n_results=1,
        include=["documents", "metadatas", "distances"],
    )

    if meta.get("flagged", 0) == 1:
        return None

    feedback_score = int(meta.get("feedback_score", 0))
    threshold = (
        FEEDBACK_TRUSTED_SIMILARITY
        if feedback_score >= FEEDBACK_TRUSTED_THRESHOLD
        else SIMILARITY_THRESHOLD
    )

    if distance <= threshold:
        return answer, entry_id
PLAIN ENGLISH

Ask ChromaDB: "find the closest stored question to this normalized query."

Only return the single best match — we just need the top hit.

Get back: the stored question, its metadata (the answer, scores), and how far away it is.

If users have flagged this entry as bad quality — skip it, even if it's a close match.

Check how many upvotes vs downvotes this entry has earned from real users.

Highly upvoted entries get a more generous threshold — they've proven themselves reliable.

Default: distance ≤ 0.15. Trusted entries (score ≥ 3): distance ≤ 0.20.

If the match is close enough — return the stored answer and its ID. Otherwise return None.

05

The Clever Tricks

Three engineering decisions that make DejaQ production-ready rather than just a prototype.

Trick 1 — don't make users wait for the filing

After generating an answer, DejaQ needs to: run Phi-3.5 to strip the tone → store the result in ChromaDB. Phi-3.5 is not instant. If users had to wait for this, every response would feel sluggish.

The fix: answer the user immediately, then hand the storage work to Celery — a task queue running on a separate process. Like a waiter handing you your food, then separately dealing with the receipt paperwork.

CODE — cache_tasks.py 
@celery_app.task(
    name="app.tasks.cache_tasks.generalize_and_store_task",
    queue="background",
    max_retries=2,
)
def generalize_and_store_task(
    self, clean_query, answer, original_query, user_id
):
    if _is_suppressed(clean_query):
        return {"status": "suppressed"}

    context_adjuster, memory = _get_services()
    generalized = context_adjuster.generalize(answer)
    memory.store_interaction(
        clean_query, generalized, original_query, user_id
    )
    return {"status": "stored"}
PLAIN ENGLISH

The @task decorator registers this function as a Celery background job.

Give it a unique name so the task queue can route it to the right worker.

"background" queue — runs on a separate process, never blocking the web server.

Auto-retry up to 2 times if it fails — network blips or model hiccups won't lose data.

First check Redis: did a user flag this query as bad recently? If so, don't re-store it.

Lazy-load the AI services — they initialize once per worker process, not once per task.

Run Phi-3.5 Mini to strip the tone and produce a neutral, reusable version of the answer.

Store the neutral answer in ChromaDB, keyed to the normalized query.

Return a status dict — Celery stores this for monitoring dashboards and debugging.

Trick 2 — users teach the cache to stretch

Not all cached entries are equally trustworthy. DejaQ tracks a feedback_score per entry and adjusts behavior dynamically:

👍
Trusted Entry (score ≥ 3)
Threshold widens from 0.15 → 0.20. The entry catches questions further away in meaning — it's proven reliable, so it earns a bigger net.
👎
Flagged Entry (score ≤ -3)
Skipped entirely during lookup. Even a perfect cosine match won't return a response that users repeatedly marked as wrong.
🚫
Suppression Flag in Redis
After a bad rating, Redis temporarily blocks re-storage of that query for 5 minutes — preventing the bad answer from immediately re-entering the cache.
🗑️
Auto-Delete (score ≤ -5)
Consistently terrible entries get deleted from ChromaDB automatically — the cache prunes itself without manual intervention.

Trick 3 — store neutral, serve with tone

This design decision is subtle but solves a real problem. Why strip tone before storing rather than just storing the raw LLM answer?

✗

Naïve approach: store the raw LLM answer

User A asked formally → stored a formal answer. User B asks casually → gets User A's stiff response. Feels wrong. You'd need to store multiple versions per question — or serve a mismatch.

✓

DejaQ's approach: store neutral, serve with tone

One neutral version stored per question. Each user gets it rewritten to their tone on retrieval. One storage cost, unlimited flexible delivery — and the stored facts are always clean.

The "master copy" pattern

Translation agencies keep one source document and produce translated versions on demand — they never archive every language. DejaQ keeps one factual master answer and "translates" it to each user's communication style on demand. Same principle, different domain.
